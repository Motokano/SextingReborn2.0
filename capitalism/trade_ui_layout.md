# 交易界面布局说明 (trade_ui_layout.md)

本文档约定交易弹窗的布局与交互要素，供实现时参照。规则与设计意图以 `basic.md` 为准。

---

## 一、基本约定

- **无「持有银两」概念**：本作中物品本身即一般等价物，不以独立货币（如银两）展示。交易以物易物为核心。
- **主内容区**：两个列表左右并排，等宽各占一侧。

---

## 二、主内容区：双列表结构

- **左侧**：玩家持有且可供交易的物品列表。
- **右侧**：交易对象可供交易的物品列表。

两列等高、可独立滚动；空态显示「空」。

### 两侧行数不一致时

- 以**行数较多的一侧**为基准确定主体结构的行数。
- **行数较少的一侧**在不足部分**留空**（即多出的行不显示物品与操作列，仅保留空行或占位，使两侧行数对齐）。
- 这样表格总行数 = max(左侧物品数, 右侧物品数)，避免因两侧长度不同导致错位或布局异常。

### 双列表下方的感受行

- 在双列表表格**下方**隔开一行（留空或分隔线），单独显示一行文案：
  - **格式**：`你觉得 [感受词]，[交易对象] 觉得 [感受词]。`
  - **[交易对象]**：当前交易对象的称呼（如行商、老者），来自 NPC 的 `sn` 或商人配置的显示名。
- **感受词**：仅从以下 **5 种** 中根据实际交易情况**各选一个**显示（玩家侧一个、对方侧一个）：
  - **不行**
  - **略亏**
  - **可以**
  - **略赚**
  - **很赚**
- 感受词由双方**给出/得到的物品的基线价值**比较得出：基线价值来自物品表中的 `base_value` 字段，仅存于表与文档，**不在任何界面向玩家展示**；实现时据此计算后从上述 5 词中择一显示。

---

## 三、列表单行结构

每一行包含两大部分：

1. **物品与数量**：`[物品名] × 数量`
2. **操作列**（在数量右侧，单独一列）：
   - **步进区**（在数量的左侧）：`[ -10 ]`、`[ -1 ]`、当前数量 `[ N ]`、`[ +1 ]`、`[ +10 ]`
   - **动作区**：`[给 N 个]` / `[要 N 个]`、`[全都给]` / `[全都要]`；「给 N 个」与「全都给」（或「要 N 个」与「全都要」）之间留 **1 个半角空格** 的间距。

### 左侧（玩家可交易物品）

- 步进：`[ -10 ]`、`[ -1 ]`、`[ N ]`、`[ +1 ]`、`[ +10 ]` 用于修改「本次要给出的数量」。
- 按钮：`[给 N 个]` 按当前 N 执行给出；`[全都给]` 给出该物品全部持有量。

### 右侧（对方可交易物品）

- 步进：`[ -10 ]`、`[ -1 ]`、`[ N ]`、`[ +1 ]`、`[ +10 ]` 用于修改「本次要拿取的数量」。
- 按钮：`[要 N 个]` 按当前 N 执行拿取；`[全都要]` 拿取对方该物品全部数量。

### 步进与范围

- 步进单位：±1、±10（共四个按钮）。
- 当前数量 N 不得超过该行物品的可用数量（左侧不超过玩家持有，右侧不超过对方持有）。
- 实现时可将 N 夹在 0 与可用数量之间，或在边界处禁用对应的步进按钮。

### 步进按钮的交互效果

- 交互者点击**任意步进按钮**（`[ -10 ]`、`[ -1 ]`、`[ +1 ]`、`[ +10 ]`）时，**仅使页面上显示的该行「操作数量」N 发生变化**。
- **不**对库存进行任何实际增减或交换操作，**不**执行任何其他指令。步进只影响界面上的数值展示，真正改变库存的动作由「给 N 个」「要 N 个」「全都给」「全都要」或后续的结算流程触发。

### 点击给/要后的待处理行

- **触发时机**：每次点击「给 N 个」「全都给」（左侧）或「要 N 个」「全都要」（右侧）时，除按当前 N 或全部数量执行一笔给/要并扣减己方当前交易库存外，在**对方列表**中需有对应的待处理行用于展示本笔数量。
- **对方列表**的含义：
  - 点击**给 N 个 / 全都给**时：在**右侧（对方可交易物品）**列表中体现该笔（新增一行或更新已有行，见下）。
  - 点击**要 N 个 / 全都要**时：在**左侧（玩家可交易物品）**列表中体现该笔（表示己方从对方处拿取、在己方列表中显示的待取项）。
- **同次交易内、同一物品只占一行；数量只改数字**：在同一次交易会话中（即每次点击「结算」按钮之前），通过步进或再次点击给/要修改某物品的待给/待要数量时，**同一物品在对应列表中只存在一行**。第一次对该物品执行给/要时新增该行；之后对该物品的数量修改**只更新该行上显示的数字**，不新增行、不重新拼接列表。实现时不应每次操作都重新生成或拼接整表行，而应复用已有行并更新其数量展示。
- **数量为 0 时隐藏行**：若某侧某物品的待处理数量变为 0，该列表中该物品的待处理行**先隐藏**（不显示），需要时再显示；行不删除，以便数量从 0 再次增加时直接更新该行数字并显示，避免重复新建行。
- **待处理行展示与扣减**：
  - **物品名样式**：该行物品名以**下划线**显示，并在名称旁**标注当前待给/待要的数目**（如「红铁块 × 3」），该数目随步进或给/要操作更新。
  - **库存扣减**：点击「给 N 个」或「全都给」后，从**玩家当前交易库存**（会话内可用于给出的数量）中扣减对应数量；点击「要 N 个」或「全都要」后，从**对方当前交易库存**（会话内可用于被拿取的数量）中扣减对应数量。
  - **按钮保留**：待处理行上仍显示该侧应有的步进区与动作按钮（「给 N 个」「全都给」或「要 N 个」「全都要」），具体交互（如是否可撤销、是否可再改数量）由实现决定，此处仅约定界面需保留按钮展示。

### 结算时的合并与库存变更

- **触发**：用户点击「结算」按钮后，本次交易会话结束并执行结算。
- **合并到列表**：结算时，所有在本会话中「给予」或「要取」的物品，按**实际交易时给予/要取的数量**，分别并入**双方已有的物品列表**（即：玩家给出的数量并入对方列表对应物品，玩家要取的数量并入己方列表对应物品）；若某物品在目标列表中尚不存在，则新增该物品及其数量。
- **库存加减**：在合并的同时，对**交易双方的库存**做对应加减：玩家给出的数量从玩家库存中扣减、加入对方库存；玩家要取的数量从对方库存中扣减、加入玩家库存。结算完成后，双方库存与列表展示应与上述规则一致。
- **隐藏行的处理**：会话内因待处理数量变为 0 而隐藏的待处理行，在以下两种情况下应**删除**（从界面与数据中移除），不再保留：**点击「结算」后**；**交易窗口被关闭时**（含未点结算即关闭、按 Esc 关闭等）。

## 四、布局模板（ASCII）

│ [物品 A]  ×  3    │  [ -10 ]  [ -1 ]  [ 1 ]  [ +1 ]  [ +10 ]    [给 1 个] [全都给]  │

```

```
